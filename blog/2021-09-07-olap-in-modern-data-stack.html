OLAP is almost 30 years old and often used in different contextess over the last 10 years. It's a huge market and there are many enterprise software in the space. OLAP is basically pre-aggregated data from your raw data and it's not for everyone. If you're working for a startup that doesn't have petabytes of data, you probably don't need OLAP because you can run ad-hoc queries on your raw data efficient enough. However; when you have petabytes of data and have tens of people who rely on this data to make decisions, you don't want to make them wait 10 minutes for a simple query. Let's breakdown the OLAP into two category:

Multidimensional OLAP (MOLAP)

Most of them have their prirority storage and compute engines and provide fast access to data in OLAP way. You store your data in your data lake or warehouse, duplicate the data into their system and slide / dice the data using their query engine. We call this type of OLAP as Multidimensional OLAP (MOLAP) because they have their own storage / query engine to run the queries. Apache Kylin, Druid, and Pinot are the open-source products in this space and well-know enterprise alternatives are Arcadia Data, [Atscale], Kyligence.

The good part is that they can quarantee to run ad-hoc queries in less than a few hundreds of milliseconds so they're pretty useful for speeding up your BI tools or building a customer facing application. On the other hand, it's not easy to setup and maintain the open-source alternatives, enterprise solutions are expensive considering you can access the data from your data warehouse as well. However; the new generation data warehouse solutions are efficient enough to answer ad-hoc queries, you don't need to run Hadoop jobs that takes hours / days to finish anymore but they also can't quarantee performance for ad-hoc queries.

1. BigQuery: BI Engine

BigQuery is a fully-managed data warehouse solution from Google. You pay for the data processed for each query that you're running so if you're running ad-hoc queries on raw data, it becomes expensive and can't really quarantee to run the queries in less than a few seconds. Luckily, you can create dimensional tables from your raw data under a schema and buy BI Engine slots that stores all the data in your dimensional tables in-memory and let you run OLAP queries efficiently for a fixed price.

2. Clickhouse: Aggregating Merge Tree

Clickhouse is an open-source OLAP database from Yandex. It supports SQL and the performance is really mind blowing given that it doesn't really use GPUs. It also offers some interesting table layouts for pre-aggregating the data storeing the intermediate state inside database tables. AggregatingmMergeTree let's you create materialized views and update them transparently when you insert data into the raw tables. It's suitable for immutable time-series data such as customer event data and it really makes sense given that Yandex initially developed Clickhouse for their Google Analytics alternative, Yandex Metrika.

4. Postgresql: Grouping sets & B-tree Indexing

Our old friend, Postgresql is actually a great solution for serving the pre-aggregated data. For raw data, you can leverage advanced techniques such as partitioning and BRIN indexes but since Postgresql stores the data in row-oriented format and doesn't have vectorized execution, the latency will not be comparable to cloud data warehouses. If you pre-aggregate your data with grouping sets and index the dimensions with B-tree, you can access the pre-aggregates metrics really fast and roll-up them efficiently as the data will be small compared to the raw data.

5. Redshift's Autorefreshing Materialized views and Snowflake's Materialized views

Redshift recently added support for materialized views that are up-to-date all the time. I didn't try it yet but the materialized views in Snowflake doesn't really work in a reliable way. They're usually not transparent and they can't quarantee the low-latency ad-hoc queries to materialized views due to fact that they actually need to update the materialized tables every time you insert data. It usually a better approach to use a transformation tool such as dbt or Airflow to transform the data and update the tables manually and transparently and use lambda views for more deterministic and reliable workloads. Most data-analysts that I talked to don't like black-boxes nowadays.

6. Materialize: Streaming Materialized Views

Materialize is not a database, it's a streaming engine. In OLAP world, near-real time workloads are usually enough to cover most of the use-cases but if you have real-time use-cases (< 1 min), Materialize's materialized views are always up-to-date and suitable for low-latency workloads. The downside is that it doesn't work inside your data warehouse, you need to setup and maintain it sepearetely.

7. [New] Firebolt: Aggregating and Join Indexes

Firebolt is a next generation data warehouse that claims to be orders of magnitude faster than Snowflake. Their real power is to be able to create aggregating, join, and partial indexing on data. While the performance will probably be comparable to Snowflake for ad-hoc queries, if you create relevant indexes in your tables, it should be much more faster as the partial computation is done before you run the query. It looks great on paper but I didn't try the product yet. (I'd love to, though!) If these new indexing types doesn't really affect the insert / update / delete performance, I believe that it's the next big thing in data warehouse industry.

Where does metric layer fit in OLAP?

Metrics layer should not duplicate the data so we fit in the ROLAP engines. Except for 5 and 6, you need to transform and pre-aggregate the data separetely. The transformation tools in the modern data stack such as dbt and Airflow helps a lot but you need to maintain all the roll-up models manually and deal with non-additive aggregates separately. Claire from Analytics Engineers Club recently wrote a great article about the problem. There is also Tristan's LookML vs dbt article that you should read about this topic. Once you define your metrics in metriql, you can create Aggregates transparently and we will create the dbt models and use them for the ad-hoc queries wherever they're applicable. Therefore we believe that when Metriql is being used by the advanced technologies that we explained under the ROLAP section, the end-solution matches the performance of MOLAP engines but don't suffer the data silo issue being a more lightweight and cheap.
